# Clinical Context input UX — согласованное решение

Статус: `ACCEPTED`  
Дата: 2026-07-10  
Связанные задачи: `HC-012b`, `HC-012b2`, `HC-033`

## Главный принцип

Пользователь не должен изучать синтаксис ввода ради заполнения медицинского профиля.

Не использовать обучение разделителям, обязательные запятые, точки с запятой или Enter как основной способ создания нескольких записей.

Целевой паттерн для состояний, симптомов, лекарств, добавок и аллергий:

```text
поиск / ввод → подсказка → выбор → отдельный визуальный chip → продолжение ввода
```

После выбора термин считается отделённым независимо от пробелов, запятых и прочих символов, введённых далее.

## Typeahead и chips

- Подсказки появляются после 2–3 символов.
- Выбранный термин немедленно превращается в отдельный редактируемый chip.
- Курсор остаётся в поле для следующего термина.
- Chip можно удалить до сохранения.
- Сохранение выполняется для набора подтверждённых chips.
- Оригинальный ввод никогда не теряется молча.

Пример:

```text
[ Головная боль × ] [ Насморк × ]
```

Для лекарств и добавок используется тот же паттерн:

```text
[ Омепразол × ] [ Магний × ] [ Витамин D3 × ]
```

## Свободный текст

Если подходящей подсказки нет, пользователь может создать отдельный chip из своей формулировки:

```text
[ ломит виски · указано пользователем × ]
```

Требования:

- свободный текст допустим;
- оригинальная формулировка сохраняется без принудительной замены;
- запись попадает в личный словарь пользователя после подтверждения;
- отсутствие совпадения со справочником не блокирует сохранение;
- сервер не отбрасывает текст и не пытается молча разрезать его на части.

Рекомендуемая модель:

```text
display_text: исходная формулировка пользователя
canonical_term_id: nullable
canonical_label: nullable
source: dictionary | personal | free_text
confirmed_by_user: true
```

## Словари

Использовать три уровня.

### 1. Базовый словарь

- симптомы;
- длительные состояния;
- лекарства;
- добавки;
- аллергены;
- синонимы и частые варианты написания;
- предустановленный тип и безопасные метаданные.

### 2. Личный словарь

- подтверждённые пользователем термины работают сразу;
- словарь привязан к пользователю и профилю/типу субъекта;
- человеческие термины не должны попадать в будущий pet-контекст;
- личная формулировка остаётся доступной независимо от канонизации.

### 3. Общий пополняемый словарь

- сырой пользовательский ввод не публикуется автоматически;
- кандидаты агрегируются анонимизированно;
- требуется порог по разным пользователям;
- требуется модерация перед публикацией;
- редкие личные формулировки не становятся общими подсказками;
- после AI-gateway AI может группировать похожие формулировки, но публикация остаётся отдельным контролируемым процессом.

## AI-нормализация

AI-нормализация реализуется только после AI-gateway с consent, единой точкой вызова и учётом стоимости.

Допустимо:

```text
болит голова → предложить связь с «головная боль»
омепрозолл → предложить «омепразол»
```

Недопустимо:

```text
жжение за грудиной → ГЭРБ
тяжело дышать → астма
головная боль → мигрень
```

Инвариант:

> Симптом никогда не превращается в диагноз автоматически.

AI может предложить нейтральную каноническую форму или исправление написания. Замена или связь применяется только после подтверждения пользователя.

Оригинальный `display_text` сохраняется даже при наличии канонической связи.

## Классификация без медицинской таксономии от пользователя

Абстрактный ярлык «текущее состояние» не использовать.

При необходимости явного выбора показывать две понятные кнопки с примерами прямо в подписи:

```text
Хроническое заболевание — например, гипертония или астма
Временный симптом — например, головная боль или температура
```

Требования:

- формулировка описывает жизненный цикл, а не требует от пользователя медицинской экспертизы;
- при выборе из словаря тип подставляется автоматически;
- для свободного текста вместо медицинской классификации задаётся вопрос о ситуации:
  - «Это у вас давно или постоянно?»
  - «Это появилось недавно и должно пройти?»
- тип можно изменить одним действием без потери истории;
- система может мягко предложить пересмотр типа, но никогда не меняет его сама;
- у каждого поля и действия есть микрокопия с конкретным примером, а не абстрактный placeholder.

Примеры микрокопии:

```text
Состояния: «Например, гипертония, астма или гастрит»
Симптомы: «Например, головная боль, температура или насморк»
Лекарства: «Например, омепразол или ибупрофен»
Добавки: «Например, магний или витамин D3»
Аллергии: «Например, пенициллин, арахис или лактоза»
```

## WhyWeAskPopover — зачем это нужно и можно ли не заполнять

У каждой секции Clinical Context должен быть доступный `WhyWeAskPopover` или эквивалентная кнопка «Зачем это нужно?».

Общие требования:

- объяснение связано с конкретной функцией продукта;
- не использовать расплывчатые формулировки вроде «для точности»;
- явно указывать: «Это необязательно»;
- отсутствие данных не блокирует использование продукта;
- поповер должен быть доступен с клавиатуры и screen reader;
- на мобильном поповер не должен перекрывать основное действие и должен закрываться явной кнопкой.

Рекомендуемая микрокопия по секциям:

### Состояния и симптомы

> Это поможет собрать историю для отчёта врачу и сопоставлять изменения самочувствия с анализами и измерениями. Заполнять необязательно.

### Аллергии и непереносимости

> Эти сведения могут попадать в emergency card и использоваться в отчёте врачу. Заполнять необязательно.

### Лекарства

> Лекарства учитываются при объяснении анализов и подготовке списка для врача. Заполнять необязательно.

### Добавки

> Добавки помогают учитывать возможное влияние на анализы и самочувствие. Заполнять необязательно.

Дополнительный инвариант:

> WhyWeAsk объясняет использование данных, но не обещает медицинский результат и не побуждает пользователя раскрывать больше, чем он хочет.

## Жизненный цикл состояний и симптомов

### Длительные состояния

Статусы:

- active;
- remission;
- inactive/ended.

Допускается возврат ранее существовавшего длительного состояния в active с сохранением истории статусов.

### Временные симптомы

Хранятся отдельными эпизодами:

- started_at;
- ended_at;
- active/ended;
- связь с повторными эпизодами одного понятия.

Кнопка «Прошло» завершает текущий эпизод. Система не завершает симптом автоматически.

Повторный симптом создаёт новый эпизод, а не продлевает старый.

## Лекарства и курсы

- Завершённый курс остаётся завершённым.
- Новый приём того же лекарства создаёт новый курс.
- Старый курс не «возобновляется» изменением его дат.
- Курсы могут ссылаться на один и тот же словарный препарат.
- Для ongoing/PRN допускается отдельная модель статуса, но история приёма не должна стираться.

Минимальные поля курса:

- medication concept / display text;
- started_at;
- ended_at nullable;
- mode: course | ongoing | prn;
- status: active | completed;
- dose/frequency nullable;
- provenance и user confirmation.

## Red flags

Для symptom episodes используется статический клиентский и серверный red-flag фильтр.

Сообщение не должно создавать ложное чувство диагностики:

> Это описание может требовать срочной медицинской помощи. Сервис не может определить тяжесть состояния. При выраженных или внезапных симптомах обратитесь в экстренную службу.

Отсутствие совпадения со статическим списком не означает, что симптом безопасен.

## Acceptance criteria для следующей реализации

- typeahead + chips для всех Clinical Context секций;
- свободный fallback отдельным chip;
- отсутствие обязательного синтаксиса разделителей;
- оригинальный текст не теряется;
- запись не превращается в диагноз;
- chips можно удалить до сохранения;
- ярлык «текущее состояние» отсутствует;
- выбор между длительным состоянием и временным симптомом сформулирован понятным языком и содержит примеры;
- у каждого поля есть микрокопия с примером;
- у каждой секции есть WhyWeAskPopover с конкретной пользой и явным «это необязательно»;
- длительные состояния и эпизоды имеют разные жизненные циклы;
- завершённый курс лекарства неизменяем как завершённый исторический курс;
- повторный приём создаёт новый курс;
- AI не требуется для базовой работы;
- личный словарь работает без AI;
- общий словарь пополняется только через анонимизированный и модерируемый контур.
