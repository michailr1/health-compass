## 1. Текущее состояние и вердикт по инциденту RLS

**Что реально работает (по коду, не по README):** backend FastAPI + PostgreSQL со схемой `health_compass` (10 прикладных таблиц), 21 миграция, прямой Google OIDC (PKCE/state/nonce, email_verified), Email Magic Links (хэшированные одноразовые токены, rate-limit в SECURITY DEFINER функциях, app-роль без прямых прав на таблицу токенов), серверные сессии, RLS с FORCE на всех таблицах, выделенная роль `health_compass_rls_definer` (NOLOGIN, BYPASSRLS) для helper-функций. Frontend — React SPA на демо-данных, к API подключён только auth.

**Инцидент `StatementTooComplex` (54001):** причина подтверждена кодом — до `FORCE RLS` владелец-migrator неявно обходил политики внутри SECURITY DEFINER функций; после `0015` политика `profile_permissions` → функция `app_can_view_profile` → чтение `profile_permissions` → та же политика → рекурсия. Исправление в `0020`/`0021` (BYPASSRLS-роль + `row_security=off` + `search_path=''` + нерекурсивные self-политики + ужесточённые bootstrap-INSERT-политики, закрывшие self-grant и self-add) — **корректно и соответствует отраслевой практике**. Альтернативы рассмотрены в `02-technical-architecture.md` §2.2; менять подход не нужно, нужно закрепить его тестами и ADR-007.

## 2. Продуктовая концепция

Health Compass = многопользовательский health-workspace: (а) архив исходных медицинских документов; (б) извлечённые и подтверждённые человеком структурированные данные; (в) динамика и связи; (г) понятные объяснения и подготовка к врачу/ветеринару; (д) семейный контур с делегированным доступом; (е) отдельный ветеринарный домен. Система никогда не ставит диагноз, не назначает лечение и не заменяет врача; каждое AI-утверждение маркировано (факт/интерпретация/рекомендация) и снабжено источниками. Подробно: сегменты и позиционирование — `12-product-strategy-and-monetization.md`.

## 3. Целевая архитектура (резюме)

Modular monolith (ADR-001): FastAPI-модули identity/tenancy/profiles/documents/labs/measurements/timeline/wearables/ai/reports/billing + воркеры (постгрес-очередь на старте, ARQ+Redis с PHASE-07), object storage (FS → S3-совместимое), ClamAV, Tesseract-OCR, pgvector для grounding, CI GitHub Actions, backup вне VPS с ежемесячным restore-тестом. Микросервисы, Kubernetes, TimescaleDB — не вводятся без доказанной необходимости. Инварианты (8 шт., обязательны для всех агентов) — `02-technical-architecture.md` §3.2. Deployment: только funti.cc, `/opt/health-compass/repo`, `health-compass-api.service`; каждая VPS-операция начинается с контрольного преамбула (hostname/IP/path/service/branch/HEAD/clean tree/alembic current/имя БД).

## 4. Human Health — функциональные направления

Для каждого направления ниже: назначение → MVP-объём → пост-MVP → ключевые риски. Модель данных — `06-data-model.md`; UX — `05-product-and-ux-specification.md`; критерии приёмки задач — листы Roadmap/Acceptance Criteria в `03-implementation-roadmap.xlsx`.

### 4.1 Медицинские анализы (PHASE-05, ядро MVP)
MVP: загрузка PDF/фото → OCR → извлечение показателей с confidence → обязательное подтверждение человеком → нормализация единиц (исходные + СИ) → референсы по полу/возрасту/лаборатории → динамика и сравнение → флаги выхода за референс → вопросы врачу. Панели MVP: ОАК, биохимия, липидный профиль, глюкоза/HbA1c, ТТГ/Т4, витамин D, ферритин; остальные (гормоны, коагулограмма, моча, микроэлементы, пользовательские панели) — v2, справочник расширяемый. Риски: OCR-ошибки числа/единицы (обязательное подтверждение + подсветка исходного фрагмента), разные нормы лабораторий (норма всегда привязана к источнику), автоматический диагноз запрещён (интерпретация — только с цитатой и дисклеймером).

### 4.2 Медицинские документы и timeline (PHASE-04)
MVP: типы «заключение/выписка/назначение/рецепт/исследование (УЗИ/МРТ/КТ/рентген/ЭКГ)/вакцинация/прочее», единая timeline с датой, источником, организацией, специалистом, статусом проверки, provenance и историей версий. v2: стоматология, госпитализации, операции, хронические заболевания как сущности, аллергии/противопоказания (аллергии — вынести в MVP-минимум как простую ручную сущность: критично для emergency card).

### 4.3 Физические параметры (PHASE-06)
MVP: вес, рост, ИМТ (вычисляемый), давление, пульс, температура, SpO2, окружности, произвольные метрики; ручной ввод ≤ 10 сек с мобильного; графики с целевыми диапазонами; связь с анализами по датам. v2: состав тела с умных весов, связи со сном/питанием/тренировками.

### 4.4 Спорт и активность
MVP: только normalized_metrics из Oura (activity, workouts). v2: собственный дневник тренировок (силовые: упражнения/подходы/повторения/вес; кардио: дистанция/темп/зоны), календарь, PR, readiness/перетренированность, ограничения при заболеваниях. Не строить свой трекер раньше, чем появится спрос: данные уже приходят из носимых.

### 4.5 Питание
v2 целиком: дневник, фото-распознавание, КБЖУ, вода/кофеин/алкоголь, интервальное голодание, аллергии. Архитектура закладывается сейчас: `foods` provider-independent (внутренний id + массив внешних ссылок провайдеров), чтобы не привязаться к одной базе продуктов.

### 4.6 Вредные привычки
v2: частота/количество/триггеры/попытки/рецидивы, деликатный неосуждающий UX, приватность по умолчанию (не видна другим членам workspace без явного гранта), связь со сном/пульсом/анализами. Без манипулятивной геймификации: только собственные цели пользователя и нейтральная статистика.

### 4.7 Генетика
v2, отдельный этап с отдельным security-ревью: VCF-хранение шифрованное и отделённое от интерпретаций; аннотации с версией базы знаний и evidence level; повторный расчёт при обновлении базы; отзыв ошибочной интерпретации; фармакогенетика — только с provenance источников. AI не объявляет вариант причиной заболевания без доказательной базы (правило в safety-фильтре). До реализации — текущий фейковый «Factor V Leiden» в bootstrap удаляется (RISK-006).

### 4.8 Сон
MVP: sleep_sessions из Oura (длительность, эффективность, фазы, пульс, HRV, температура) + субъективная оценка. v2: бессонница/храп/риски апноэ (опросники), связи с кофеином/алкоголем/тренировками/лекарствами.

### 4.9 Oura и гаджеты (PHASE-07)
MVP: Oura OAuth → sleep/readiness/activity/heart rate/HRV/temperature/workouts/tags; универсальный ingestion-слой сразу (connections → sync_runs с курсором → raw_events идемпотентно → normalized_metrics со schema_version и provenance) — чтобы Apple Health/Garmin/Fitbit/Whoop/Withings/CGM подключались без переделки ядра. Ошибки провайдера: retry с backoff, rate-limit-учёт, reconciliation по raw.

### 4.10 Прочие human-функции из §7 ТЗ
MVP-минимум: аллергии, лекарства с расписанием и фактическим приёмом, вопросы врачу, экспорт данных, deletion requests, consent management. v2: дневник симптомов/настроение/стресс, вакцинация, семейный анамнез, чек-апы, женское здоровье (цикл/беременность — с повышенной приватностью), боли/травмы/реабилитация, follow-up, emergency card (см. §5), наследование доступа, стоматология/зрение/слух.

## 6. Pet Health

Архитектурное решение (ADR-018): **раздельные доменные сущности**, не `profile_type`. Общие: users, workspaces, permissions-механика, files, timeline-инфраструктура, notifications, audit, exports, AI-инфраструктура (но не промты). Строго раздельные: справочники норм (`pet_lab_reference_ranges` по виду/породе/возрасту/полу), дозировки (`dose_per_kg`), AI-промты и retrieval-индексы, emergency-правила. Причина: единственная надёжная защита от «человеческая норма применена животному» — невозможность такого JOIN на уровне схемы.

Pet MVP (PHASE-11, после human MVP, не блокирует его): карточка питомца (вид/порода/пол/возраст/вес/стерилизация/микрочип/клиника), vet timeline (визиты/диагнозы/назначения/документы), анализы с видовыми референсами, вес и BCS, вакцинации + противопаразитарные обработки с напоминаниями, лекарства с дозой на кг и фактическим приёмом, семейный pet-dashboard (кто требует внимания/кому дать лекарство/просроченные вакцины). v2: питание/корма, поведение (лоток/вылизывание/хромота — видоспецифичные чек-листы кошка/собака), устройства (GPS-ошейники/кормушки/умные лотки), режимы «пожилой питомец»/«послеоперационный», ownership-передача. Veterinary AI: объясняет анализы/суммирует историю/готовит вопросы/находит изменения веса-воды-аппетита-активности; НЕ ставит диагноз, НЕ рассчитывает дозы, НЕ применяет человеческие нормы; красные флаги (отказ от воды, затруднённое дыхание, судороги, невозможность мочеиспускания, отравление и т.д.) → немедленный экран «к ветеринару», без AI-рассуждений. Отдельные ADR (ADR-018), wireframe (стр. Pet в `05-portal-wireframes.pdf`), war games WG-024…027, промты (раздел Pet в `09-agent-prompts/`).

## 7. Искусственный интеллект и AI safety

Архитектура (PHASE-08): оркестратор `ai_runs` → retrieval (pgvector, фильтр по profile_id/pet_id ДО поиска) → генерация с обязательными citations (`ai_evidence`) → safety-фильтр (диагнозы/дозировки/отмена лечения → блок + `ai_safety_events`) → маркировка факт/интерпретация/рекомендация. Возможности по этапам: v1 — объяснение анализов и отчёты с цитатами; v1.1 — вопросы врачу, суммаризация истории; v2 — анализ взаимосвязей с явной шкалой (совпадение → корреляция → гипотеза → возможная причинность → доказанная связь; выше «гипотезы» — только при наличии внешних источников), план действий с подтверждением пользователем и сроком актуальности рекомендаций. Пайплайн документов (20 шагов из ТЗ §9.3) реализуется в PHASE-04/05 c provenance и версией модели на каждом извлечении. Safety-контур целиком (prompt-injection-защита, недоверенные документы, cross-profile/human-pet изоляция, data minimization и consent для внешнего LLM, удаление embeddings, risk-scoring ответа, аудит) — `07-security-and-threat-model.md` §2.4; ADR-012/ADR-020. Ни один реальный AI-ответ не показывается пользователю до приёмки PHASE-08 safety-чеклиста.

## 8. UX (резюме)

Sitemap, флоу, дизайн-токены, компонентная карта, критерии приёмки — `05-product-and-ux-specification.md`; 8 wireframe-экранов — `05-portal-wireframes.pdf`. ## 9. Монетизация (резюме)

Free полезен (1 human + 1 pet, timeline и ручной ввод без лимитов, квоты на OCR/AI/хранилище); первым запускается Personal, затем Family; Pet — аддоном; запрещённая монетизация зафиксирована жёстким списком (никакой продажи данных, тревоги ради продаж, платного экспорта). Полностью: `12-product-strategy-and-monetization.md`, `13-pricing-and-entitlements.xlsx`, `14-unit-economics.xlsx`.

## 11. Приоритизация

## 14. Итоговые рекомендации

**10 важнейших архитектурных решений:** 1) modular monolith без микросервисов; 2) RLS с FORCE + BYPASSRLS-definer-роль как единый механизм изоляции (закреплено ADR-007); 3) identity=(provider,subject), email не ключ; 4) одна транзакция на запрос с транзакционным RLS-контекстом; 5) раздельные human/pet домены на уровне схемы; 6) слои факт→извлечение→подтверждение→интерпретация с provenance; 7) universal wearable ingestion (raw immutable + normalized rebuildable); 8) файлы в object storage, не в БД; 9) очередь на PostgreSQL до доказанной потребности в Redis; 10) AI строго retrieval-grounded с citations и safety-фильтром до первого пользователя.

**10 важнейших продуктовых решений:** Free полезен; activation = подтверждённый анализ ≤5 мин; позиционирование MVP — архив+динамика+подготовка к врачу; Personal первым, Family вторым; Pet аддоном после human MVP; отчёт врачу — триггер конверсии; никакой тревожности ради вовлечения; экспорт всегда бесплатен; демо — только в примере-профиле; billing только после PHASE-12-инфраструктуры.

**10 опаснейших технических ошибок (не допустить):** правка кода на production; миграция без backup и downgrade; новая таблица без RLS-политик в той же миграции; commit посреди запроса (сброс RLS-контекста); объединение аккаунтов по email; хранение файлов в БД; синхронный OCR в HTTP-запросе; секреты в логах/истории shell; деплой frontend раньше backend/миграций; работа агента на de.funti.cc.

**10 опаснейших медицинских и AI-ошибок:** фейковые данные в реальном профиле (уже случилось — RISK-006); человеческие нормы/дозы для животных; AI-диагноз или отмена лекарств; ответ без источников; cross-profile утечка в retrieval; выполнение инструкций из загруженного документа; необратимая перезапись исходного извлечения; genetic-интерпретация без evidence level; red-flag симптом, ушедший в «болтовню» вместо экрана экстренной помощи; отправка данных во внешний LLM без согласия.

**Что нельзя в MVP:** генетика, veterinary AI-советы, B2B, marketplace, corporate wellness, автоматические причинные выводы, собственный трекер тренировок. **Заложить сейчас, реализовать позже:** consents, audit_log, deletion_requests, entitlements-ключи в коде, pet-домены в схеме (пустые), profile_relations, outbox. **Всегда бесплатно:** просмотр своих данных, ручной ввод, экспорт, emergency card, timeline. **Разумно монетизировать:** OCR/AI-квоты, хранилище, семейные роли, расширенная аналитика, генетика (v2). **Нельзя монетизировать:** данные, тревогу, экстренную информацию, экспорт. **Первый тариф:** Personal. **Отдельный Pet-тариф:** не сейчас — аддон. **Family plan:** да, вторым. **Слишком дорого для Free:** массовый OCR, внешние LLM без лимита, генетика, большие файлы. **Гипотезы до billing:** activation ≤5 мин достижим; retention D30 > 20% на анализах+Oura; семьи реально приглашают ≥1 участника; владельцы питомцев ведут вес/вакцинации ≥ 1 раз/нед.
