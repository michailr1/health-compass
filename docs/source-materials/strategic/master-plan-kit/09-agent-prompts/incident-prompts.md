# Промты: Incident agent

Общие правила: сначала диагностика и фиксация фактов, изменения — только по runbook; секреты не выводить; каждый инцидент завершается отчётом в docs/incidents/YYYY-MM-DD-<slug>.md (симптомы, причина, воздействие, таймлайн, исправление, предотвращение, ссылки на RISK/WG).

## Live log monitoring
journalctl -u health-compass-api -f с фильтрами ERROR|Traceback|SQLSTATE; при всплеске 5xx — снять последние 200 строк, статус сервиса, загрузку CPU/RAM/диска, число соединений PostgreSQL (pg_stat_activity), НЕ перезапускать сервис без деградации health-эндпойнта.

## Auth failure (пользователи не могут войти)
Проверить по порядку: /health эндпойнт; исходящую доступность accounts.google.com (OIDC discovery); срок действия client secret; SMTP-доступность (для magic links); журнал 401/400 по /auth/callback (state mismatch → проблемы cookie/прокси, проверить заголовки Apache и path cookie /health/api). Классифицировать: Google outage / конфиг / cookie / код. Google outage → включён ли email-вход (fallback уже реализован) — сообщить владельцу, ничего не менять.

## Migration failure
СТОП после первой ошибки alembic; зафиксировать SQLSTATE и мигрционный файл; проверить, частично ли применена миграция (alembic current + ручная проверка объектов из миграции через psql \d); НЕ выполнять downgrade автоматически, если миграция не транзакционна (CREATE INDEX CONCURRENTLY и т.п.); восстановить из pre-deploy backup только по решению владельца; отчёт с точным состоянием схемы.

## RLS incident / подозрение на cross-user leak
Немедленно: зафиксировать запрос/скриншот; определить затронутые таблицы и политики (psql: \d+ таблица, pg_policies); проверить наличие контекстных GUC в подозрительном пути кода; воспроизвести на staging двумя тестовыми пользователями; если утечка подтверждена — по решению владельца перевести сервис в maintenance (остановить unit), т.к. конфиденциальность важнее доступности; после фикса — обязательный пост-инцидентный тест в HC-002-пакете и запись в audit.

## Wearable sync failure
Смотреть wearable_sync_runs: error, курсор; классифицировать expired token (→ пометить connection reauth_required, уведомить пользователя) / rate limit (→ backoff, не сбрасывать курсор) / schema change провайдера (→ raw события сохранены? если да — инцидент некритичен, править нормализацию) / наша ошибка. Курсор назад двигать только с идемпотентной защитой по provider_event_id.

## OCR failure
Проверить очередь (глубина, зависшие задачи), ресурсы воркера, конкретный документ (повреждённый файл?); повреждённые файлы помечать failed с причиной, не ретраить бесконечно; массовый сбой → остановить воркер, разобрать первую ошибку.

## AI safety incident (опасный совет/утечка в ответе)
Зафиксировать ai_run_id, промт-версию, evidence; заблокировать соответствующий промт-шаблон флагом; проверить, единичный ли случай (поиск по ai_safety_events); отчёт владельцу в течение того же дня; фикс — только через изменение фильтра/промта с тестом, воспроизводящим кейс.

## Disk full
Определить потребителя (du -x верхнего уровня; частые кандидаты: логи journald, /var/backups, каталог загрузок); безопасно: journalctl --vacuum-size=500M, ротация старых бэкапов ПОСЛЕ проверки наличия свежего внешнего; НЕ удалять пользовательские файлы и WAL; после освобождения проверить, не перешёл ли PostgreSQL в read-only.

## Database outage
systemctl status postgresql; журнал PostgreSQL; при OOM-kill — проверить настройки памяти; при повреждении — НЕ выполнять pg_resetwal, восстановление только из backup по runbook с решением владельца; API держать выключенным до консистентности БД.

## Rollback
Всегда парой: git checkout <previous_head> + alembic downgrade <previous_migration_head> (в обратном порядке применения); если downgrade невозможен (данные уже записаны в новую структуру) — restore из pre-deploy backup с явной потерей данных за окно — только владелец принимает это решение; после rollback — полный smoke и отчёт.
