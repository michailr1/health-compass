# Health Compass — AI product and safety baseline

Версия: 1.0  
Дата: 2026-07-09  
Статус: approved target design / planned implementation

Этот документ расширяет `SECURITY-INVARIANTS.md` для AI-контура. Он фиксирует обязательные продуктовые и технические ограничения из материалов Fable этапа 3.

## 1. Назначение AI-контура

Планируемые функции:

- объяснение трендов и лабораторных показателей;
- поиск корреляций;
- извлечение данных из документов;
- периодические отчёты;
- отчёт врачу;
- отдельный будущий отчёт ветеринару;
- безопасные планы действий без назначения лечения;
- ответы по данным активного профиля.

## 2. Обязательные инварианты

- AI не ставит диагноз.
- AI не назначает, не отменяет и не меняет лечение.
- AI не рассчитывает дозировки.
- Содержательный ответ без evidence запрещён.
- Retrieval фильтруется по `profile_id` или `pet_id` до поиска.
- Human и Pet используют разные prompts, индексы, нормы и safety rules.
- OCR-данные требуют подтверждения пользователя до публикации как факта.
- Документы считаются недоверенным вводом, а не инструкциями модели.
- Внешний LLM используется только при отдельном согласии.
- Red flags приводят к детерминированному urgent-help banner без свободной AI-диагностики.
- Недостаток данных должен приводить к явному отказу или запросу уточнения, а не к догадке.

## 3. Evidence и provenance

Каждый содержательный AI-ответ должен быть связан с `ai_evidence`.

Минимально сохраняются:

- идентификатор профиля;
- идентификатор источника;
- тип источника;
- дата или период данных;
- версия норм/справочника;
- часть ответа, которую подтверждает источник;
- confidence;
- модель и версия prompt.

В UI Fact, Interpretation и Recommendation отображаются раздельно.

При конфликте источников ответ должен:

1. явно показать конфликт;
2. не выбирать источник молча;
3. объяснить, какие данные требуют подтверждения;
4. предложить безопасный следующий шаг.

## 4. OCR и документы

- Извлечённые значения получают статус `needs_review`.
- Low-confidence fields нельзя массово подтверждать без явного действия пользователя.
- Пользователь видит оригинальный фрагмент, значение, единицу, дату и источник.
- Prompt injection внутри документа не исполняется.
- Попытки injection логируются как `ai_safety_event` без сохранения лишнего медицинского содержимого.
- Повторная обработка не должна создавать дубли без идемпотентного ключа.

## 5. Red flags

Red flags определяются утверждёнными правилами и версионируемыми медицинскими справочниками, а не свободным рассуждением модели.

Поведение:

- показать срочность и понятную рекомендацию обратиться за помощью;
- не скрывать uncertainty;
- не формулировать диагноз;
- не генерировать дозировки;
- сохранять версию правила и источник;
- не использовать красный для обычных предупреждений или AI-гипотез.

## 6. Consent и внешние модели

До отправки данных внешнему LLM требуется отдельное, отзывное consent.

Без consent система должна:

- не отправлять профильные или медицинские данные внешнему провайдеру;
- использовать только локальные/детерминированные функции, если они доступны;
- ясно объяснить недоступность AI-функции;
- не включать внешний AI автоматически после обновления условий.

Отзыв consent прекращает новые внешние вызовы. Retention и удаление уже созданных derived data регулируются политикой lifecycle.

## 7. Privacy-minimized logging

`ai_runs` может содержать:

- model/provider;
- prompt version;
- latency;
- token/cost metadata;
- safety outcome;
- технический error code;
- ссылки на внутренние идентификаторы evidence.

Не допускается сохранять в обычные логи:

- полный prompt с медицинскими данными;
- полный ответ модели;
- access tokens;
- session tokens;
- OAuth secrets;
- SMTP credentials;
- исходные документы;
- необязательные медицинские значения.

## 8. Human/Pet separation

Для Human и Pet обязательны отдельные:

- system prompts;
- retrieval indices;
- norm dictionaries;
- red-flag rules;
- report templates;
- evaluation datasets;
- safety tests.

Активный контекст должен быть визуально виден пользователю до отправки вопроса.

## 9. Transparency и feedback

Пользователь должен иметь возможность:

- открыть evidence;
- увидеть источник и дату;
- отметить полезность или ошибку;
- сообщить о небезопасном ответе;
- понять, где факт, а где интерпретация;
- увидеть, что данных недостаточно.

Модель и версия prompt могут показываться в расширенных деталях или audit view.

## 10. Минимальные тесты до включения AI

- ответ без evidence блокируется;
- retrieval не возвращает данные другого профиля;
- Human запрос не получает Pet данные и наоборот;
- prompt injection из документа не меняет системные правила;
- dose-calculation запрос получает безопасный отказ;
- конфликтующие источники отображаются явно;
- low-confidence OCR не становится confirmed автоматически;
- red flag выдаётся детерминированно и без диагноза;
- отказ consent блокирует внешний LLM;
- логи не содержат медицинских значений и секретов.

## 11. Planned data model

Минимальные сущности AI-контура:

- `ai_runs`;
- `ai_evidence`;
- `ai_feedback`;
- `ai_safety_events`;
- versioned prompt registry;
- versioned medical/veterinary rule registry.

Точная схема утверждается отдельным ADR и миграцией вместе с RLS policies и двухпользовательскими негативными тестами.

## 12. Stop conditions

AI-функция не выпускается при:

- отсутствии evidence enforcement;
- возможности cross-profile retrieval;
- смешении Human/Pet контуров;
- логировании чувствительных данных;
- отсутствии consent enforcement;
- генерации дозировок или назначения лечения;
- red-flag logic, зависящей только от свободного ответа модели;
- отсутствии негативных safety tests.
