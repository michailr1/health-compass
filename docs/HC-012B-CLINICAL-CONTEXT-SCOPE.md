# HC-012b — Clinical Context MVP

Статус: implementation started  
Ветка: `feat/clinical-context-mvp`

## Почему нужен отдельный HC-012b

Исходный HC-012 из roadmap описывал только `date_of_birth` и `sex`. Эти поля, а также `height_cm`, timezone, вес, readiness и health-data consent уже реализованы. Следующий полезный слой профиля — минимальный клинический контекст, который влияет на безопасность интерпретаций, документов, лабораторных показателей и будущих AI-рекомендаций.

Master plan относит к MVP-минимуму аллергии и лекарства. Диагнозы, симптомы, семейный анамнез, вакцинация, питание и emergency card остаются отдельными последующими блоками.

## Scope этого PR

### 1. Аллергии

Ручная сущность профиля:

- аллерген / вещество;
- реакция свободным текстом;
- тяжесть: `unknown`, `mild`, `moderate`, `severe`;
- статус: `active`, `resolved`, `entered_in_error`;
- дата начала, если известна;
- заметка;
- provenance `manual`;
- created/updated timestamps.

### 2. Текущие лекарства

Ручная сущность профиля:

- название препарата;
- доза свободным текстом;
- схема/частота свободным текстом;
- показание/зачем принимается — необязательно;
- статус: `active`, `paused`, `stopped`, `entered_in_error`;
- даты начала/окончания;
- заметка;
- provenance `manual`;
- created/updated timestamps.

### 3. API

Profile-scoped CRUD:

- list/create/update allergy;
- list/create/update medication;
- физическое удаление не используется: ошибка ввода отмечается статусом `entered_in_error`;
- все операции требуют активного `health_data_processing` consent;
- view/edit проверяются PostgreSQL RLS, а не только Python-кодом.

### 4. UI

На странице «Профиль здоровья»:

- секция «Аллергии»;
- секция «Текущие лекарства»;
- компактное добавление с телефона;
- редактирование/остановка без модального лабиринта;
- пустые состояния;
- явная подпись, что сведения введены пользователем и не являются врачебным назначением;
- severe active allergy визуально выделяется.

### 5. Clinical readiness

Профиль должен различать:

- контекст не заполнен;
- пользователь явно указал «аллергий нет»;
- есть активные аллергии;
- лекарства не заполнены;
- пользователь явно указал «постоянных лекарств нет»;
- есть активные лекарства.

Для этого нужны отдельные profile-level timestamps/flags подтверждения отсутствия, а не вывод «нет записей = нет аллергий».

## Security invariants

- новые таблицы: `ENABLE ROW LEVEL SECURITY` + `FORCE ROW LEVEL SECURITY`;
- app-role не получает обход RLS;
- политики используют существующие `app_can_view_profile` / `app_can_edit_profile`;
- индексы по `profile_id`;
- никаких SECURITY DEFINER CRUD-функций без необходимости;
- consent проверяется транзакционно;
- audit metadata не содержит лишних медицинских текстов;
- никакого автоматического диагноза или изменения лекарств AI-системой.

## Не входит

- справочник лекарств и взаимодействий;
- расписание и фактические приёмы;
- push-напоминания;
- диагнозы/conditions;
- семейный анамнез;
- emergency QR card;
- OCR-извлечение из назначений;
- AI-рекомендации;
- питание и фото еды.

## Acceptance

1. Пользователь с edit-доступом создаёт и обновляет аллергии и лекарства.
2. Viewer видит записи, но не меняет их.
3. Чужой пользователь не видит записи даже прямым SQL/API-запросом.
4. Без active consent создание/изменение блокируется.
5. Пустой список не трактуется как подтверждённое отсутствие.
6. Можно явно подтвердить «аллергий нет» и «постоянных лекарств нет».
7. Severe active allergy заметна на странице профиля.
8. Все backend/frontend/migration/RLS tests зелёные.
9. Production не меняется до отдельного rollout-решения.
