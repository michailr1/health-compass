# Health Compass — Progressive Health Intake

Версия: 1.0  
Дата: 2026-07-09  
Статус: approved target design / planned implementation

Этот документ фиксирует принятое решение по PHASE-02.5. Источники: материалы Fable `02.5-health-intake-spec.md`, `02.5-intake-fields.xlsx`, `02.5-intake-wireframes.pdf`, а также решения владельца проекта.

## 1. Основное решение

PHASE-02.5 не является блокирующей анкетой перед первым анализом. Health intake реализуется как прогрессивно наполняемый паспорт здоровья.

Целевой путь:

```text
Login
→ минимальный onboarding
→ Empty Dashboard / первое полезное действие
→ добровольное дозаполнение Health Profile
→ точечные контекстные вопросы при интерпретации
→ подтверждённый импорт фактов из документов
```

Инварианты:

- активация до первого полезного действия не должна блокироваться длинной анкетой;
- все медицинские поля, кроме имени профиля и необходимых согласий, опциональны;
- каждое поле сопровождается объяснением, зачем оно нужно;
- отсутствие данных снижает точность контекста, но не блокирует продукт;
- intake не является самодиагностикой;
- AI не изменяет профиль без подтверждения пользователя.

## 2. Поле «Пол»

Для MVP используется одно поле `sex`, без разделения пола и гендера.

Допустимые значения:

```text
male
female
not_specified
```

Правила:

- пользовательский label: «Пол»;
- поле опционально;
- используется для выбора половозрастных лабораторных референсов;
- при отсутствии значения применяются общие референсы лаборатории с предупреждением о сниженной точности;
- значение можно изменить или удалить;
- AI не выводит диагноз на основании этого поля.

## 3. Уровни intake

### Уровень 0 — минимальный onboarding

- имя профиля — обязательное;
- дата рождения — опционально;
- пол — опционально;
- обязательные согласия;
- отдельное согласие на внешний LLM, если функция используется.

Должна быть доступна команда «Пропустить и заполнить позже».

### Уровень 1 — контекстная точность

Запрашивается только тогда, когда конкретный ответ меняет интерпретацию:

- беременность/лактация;
- рост;
- иные клинические параметры, необходимые конкретному валидированному правилу.

Этническая принадлежность не входит в обычный MVP intake и не участвует в общем показателе полноты. Она может быть запрошена только будущим конкретным валидированным правилом с явным объяснением.

### Уровень 2 — клинический контекст

- хронические состояния;
- аллергии;
- текущие лекарства;
- текущие добавки, если они влияют на интерпретацию.

Эти данные вводятся вручную или добавляются из документа только после подтверждения пользователя.

### Уровень 3 — образ жизни

Не входит в первый scope PHASE-02.5:

- курение и алкоголь;
- активность;
- питание;
- сон;
- стресс.

Они собираются позже в соответствующих модулях и не должны превращать Health Profile в стену-опросник.

## 4. Точки сбора

### Onboarding

Только уровень 0. Цель — быстро создать профиль и вывести пользователя к первому полезному действию.

### Health Profile

Планируемый маршрут:

```text
/p/:profileId/health-profile
```

Экран содержит:

- основные данные;
- клинический контекст;
- состояния;
- аллергии;
- лекарства и добавки;
- provenance;
- autosave;
- удаление и редактирование;
- объяснение «Почему мы спрашиваем?».

### Контекстный дозапрос

Компонент `IntakePromptCard` появляется только при реальной необходимости, например на экране динамики показателя или в AI-объяснении.

Действия:

- ответить и сохранить в профиль;
- применить только к текущему анализу;
- не сейчас;
- открыть объяснение.

### OCR Review

Если документ содержит аллергию, состояние, лекарство или добавку, система предлагает добавить факт в профиль. Без явного подтверждения профиль не меняется.

## 5. Полнота профиля

Общий процент допускается только как мягкий ориентир без красного цвета, блокировок и давления.

Основной UX — контекстная готовность:

```text
Профиль готов для возрастных и половых референсов
Для интерпретации ферритина не хватает 1 уточнения
```

Чувствительные и нерелевантные поля не должны искусственно снижать полноту профиля.

## 6. Навигация и компоненты

Health Profile находится в группе «Профиль» desktop sidebar. На мобильном — в разделе «Ещё». Он не конкурирует с основным CTA «Загрузить анализ».

Новые компоненты:

- `HealthProfileForm`;
- `IntakePromptCard`;
- `CompletenessMeter`;
- `WhyWeAskPopover`.

Переиспользуются:

- `ConsentDialog`;
- `DomainBadge`;
- `ProvenancePopover`;
- `UndoSnackbar`;
- `PermissionGate`.

Human accent: `#0E7490`. Красный применяется только для подтверждённых red flags и destructive actions.

## 7. Данные, provenance и безопасность

Для каждого значения сохраняются:

- значение;
- источник: `manual`, `document`, `device`, `import`;
- статус подтверждения;
- время создания и изменения;
- actor/user;
- profile ownership;
- при необходимости связь с исходным документом.

Все таблицы PHASE-02.5 должны иметь RLS и negative cross-user tests.

AI получает только минимально необходимый intake-контекст и только при соответствующем consent. Свободный текст пользователя не считается подтверждённым диагнозом.

## 8. Первый scope реализации

В первый implementation slice входят:

1. редактирование имени профиля, даты рождения и пола;
2. экран Health Profile;
3. базовые сущности состояний, аллергий, лекарств и добавок;
4. provenance и audit trail;
5. autosave и undo;
6. мягкая contextual readiness;
7. один reusable `IntakePromptCard`;
8. API contracts, permissions, RLS и negative tests;
9. подготовка интерфейса подтверждённого импорта из OCR.

Не входят:

- большая обязательная анкета;
- полная emergency card;
- семейный анамнез;
- lifestyle-модули;
- автоматическая диагностика;
- автоматическое добавление фактов из OCR;
- Pet intake.

## 9. Связь с этапами

```text
PHASE-01/02 — Auth MVP ✅
PHASE-02.5 — Progressive Health Intake
PHASE-03 — Documents, OCR, Labs, Dynamics, AI Evidence, Doctor Report
```

PHASE-02.5 является связующим UX- и data-layer, а не отдельным изолированным медицинским доменом.
